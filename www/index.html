WHAT IS OPEN SOURCE?

Open Source Software is just like another software, however, you get free access to its source code, you may use, modify and redistribute it. The open source software is free. Developers from all over the world may contribute to a project, improvise on it and reditribute for the benefit of all.

Contribution form a diversity ebables OSS to integrate a variety of innovative ideas for the software. With so many eyeballs scanninng through the code, it becomes difficult for a bug to escape!

Open Source Software is not a new concept. It has been in use since mid 1990s. Today, it is used by a large number of powerful organizations for whom security is a priority.

LICENCE

Open Source Softwares are licenced, however the OSS licence is significantly different from the proprietary software licence. Open source licences are essentially terms of use. Open source software licenses promote collaboration and sharing because they allow other people to make modifications to source code and incorporate those changes into their own projects. Some open source licenses ensure that anyone who alters and then shares a program with others must also share that program's source code without charging a licensing fee for it. In other words, computer programmers can access, view, and modify open source software whenever they like—as long as they let others do the same when they share their work.

There are a variety of licence models for open source, where each licence model has specific terms for the use and modification of code. For this reason, it is important to understand both the specifics of the open source licence in question and how the Department intends to use and redistribute any modified OSS.

Examples of open-source licenses include Apache License, BSD license, GNU General Public License, GNU Lesser General Public License, MIT License, Eclipse Public License and Mozilla Public License.

WHY OPEN SOURCE?

Open Source Softwares are reliable and more secure. A less secure software is a lso a software bug. These bugs are usually addressed with speedy fixes wherever possible and Open Source contributors from all over work to fix such bugs. Hence, severe defects tend to be fixed within hours of their being detected. This is possible only because of the availability of the source code. Able developers who discover a bug also fix it and then report it.

By virtue of being distributed by policy, open source projects necessitate modern software development workflows. These workflows are electronic (meaning process is naturally captured and exposed), asynchronous (meaning decisions are time and location agnostic), and lock-free (meaning contributors can rapidly experiment without prior approval). Open source software users do not need to wait for years to deploy a solution. Open source software can be “test driven” prior to procurement, and is particularly suitable for inter-agency collaboration, rapid prototyping and experimentation.

All modern organizations, and increasingly more traditional organizations are adapting Open Source softwares. Five-or-ten years down the line, the technology landscape will be less closed and more collaborative.

Open Source Software are often developed by collaborative communities who volunteer their time and expertise and are coordinated by fewer paid programmers. Open Source Contribution is thus seen as more of a hobby. These do not have a per-seat licensing cost.

How can You Contribute to Open Source?


Open Source Softwares are a result of contributions made by developers from all over the world. These work well only because of the contribution to creation and maintenance of the softawres. Anyone can contribute to Open Source. In order to contribute, you need not be a programming genius with unlimited free time to contribute. Successful open source projects thrive on a wide variety of contributions from people with all levels of coding skills and commitment.

How to get started?

First and foremost, choose a programming language you like and are most comfortable with. Once you are done selecting a programming language, choose a project to contribute to. Selection of a project also depends on your interest. You may choose a project you are using the most or the one that interests you the most or you may even go by the trends. Check Trending Projects

Let's assume you choose organization X, go to thier Wiki check if you know the technologies that they use. Read all relevant information about how to contribute in the project website. Most of the FOSS Organizations have a lot to help a newbie to contribute to their Software. Clone the source code. List down the bugs you find and report. In case you are unaware of the technologies they use, learn them now!

Whenever you explore a project, the first thing you look for is its documentation. But not all developers are eager to write one. However, the documentation is necessary for the success of the project.
In the documentation, one looks for examples on how to use. It is easy to come up with examples if you have been using the software for a while. You may write small pieces of your working codes, or in case of applications, you may share a few screenshots of common usage of a tool.

Answering questions on Stack Overflow where programmers help other programmers, is also a way to contribute to Open Source. If you are experience with a given piece of Open Source Software, you will probably be able to help people who are newer to it.

Test the code while it is being developed. Taking a daily or weekly snapshot, installing it, and giving feedback is very helpful and welcome. This results in fixing of numerous bugs and helps improve the software.

In case you are a designer, you may help improve the design of the website or design a logo for the project. Many developers may not be very good with graphics. Updating sites, writing test suites to be run on various platforms are some ways to contribute to Open Source Projects.

WHAT IS IT?


To understand GitHub, you must first have an understanding of Git. Git is an open-source version control system that was started by Linus Trovalds - the same person who created Linux. So, Git is a “version control system,” what does that mean? When developers are creating something (an application, for example), they are making constant changes to the code and releasing new versions, up to and after the first official (non-beta) release.Version control systems keep these revisions straight, and store the modifications in a central repository.

Git is a command-line tool, but the center around which all things involving Git revolve – effectively, the Hub, is GitHub.com, where developers can store their projects and network with likeminded people.

Although GitHub is a Git repository hosting service, but it adds many of its own features. While Git is a command line tool, GitHub provides a Web-based graphical interface. It also provides access control and several collaboration features, such as a wikis and basic task management tools for every project.

WHY GitHub?


Because it’s a social network that has completely changed the way we work. Each user on GitHub has their own profile, which can acts like a resume of sorts, showing your past work and contributions to other projects via pull requests. Simply by being a member, you can brush elbows with the likes of Google and Facebook. Before GitHub existed, major companies created their knowledge mainly in private. But when you access their GitHub accounts, you’re free to download, study, and build upon anything they add to the network.

Still why use something like Git? Say you and a coworker are both updating pages on the same website. You make your changes, save them, and upload them back to the website. So far, so good. The problem comes when your coworker is working on the same page as you at the same time. One of you is about to have your work overwritten and erased.

A version control application like Git keeps that from happening. You and your coworker can each upload your revisions to the same page, and Git will save two copies. Later, you can merge your changes together without losing any work along the way. You can even revert to an earlier version at any time, because Git keeps a “snapshot” of every change ever made.

HOW TO USE IT?


Before we go on learning how the GitHub workflow is lets learn some basic terminologies associated with GitHub :-

Command Line : The computer program we use to input Git commands. You type text-based commands, known as prompts, into the screen, instead of using a mouse.

Repository : A directory or storage space where your projects can live. Sometimes GitHub users shorten this to “repo.” It can be local to a folder on your computer, or it can be a storage space on GitHub or another online host. You can keep code files, text files, image files, you name it, inside a repository.

Version Control : Basically, the purpose Git was designed to serve. When you have a Microsoft Word file, you either overwrite every saved file with a new save, or you save multiple versions. With Git, you don’t have to. It keeps “snapshots” of every point in time in the project’s history, so you can never lose or overwrite it.

Commit : This is the command that gives Git its power. When you commit, you are taking a “snapshot” of your repository at that point in time, giving you a checkpoint to which you can re-evaluate or restore your project to any previous state.

Branch : How do multiple people work on a project at the same time without Git getting them confused? Usually, they “branch off” of the main project with their own versions full of changes they themselves have made. After they’re done, it’s time to “merge” that branch back with the “master,” the main directory of the project

Issues : Most software projects have a bug tracker of some kind. GitHub’s tracker is called Issues, and has its own section in every repository.
